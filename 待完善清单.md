## 一、安全

### G1 · 租户管理员可越权提权为超管 ⚠️ 真实漏洞

- **现象**：`RoleService.create` 对角色 `code` 无保留字校验；`StpInterfaceImpl` 只要角色码含 `SUPER_ADMIN` 即返回通配 `*` 并绕过套餐边界。租户管理员可在本租户新建 `code=SUPER_ADMIN` 的角色 → 分配给自己 → 取得 `*`。
- **影响**：因 `tenant` 表在 `IGNORE_TABLES`（全局、不按租户过滤），提权后可列出/操作**所有租户**，属基座级越权。
- **建议**：系统角色码（`SUPER_ADMIN` 等）设为保留字，禁止租户内创建/复用；超管判定改为绑定平台租户或独立标志位，而非仅凭角色码字符串。
- **涉及**：`system/service/RoleService.java`、`system/StpInterfaceImpl.java`、`system/RbacConstants.java`
- **优先级**：P0 ｜ **状态**：待评估

### G3 · 登录无防爆破

- **现象**：`AuthService.login` 无失败计数、锁定、验证码。
- **建议**：失败 N 次锁定/退避（Redis 计数），可选滑块/图形验证码；记录失败来源 IP。
- **涉及**：`system/service/AuthService.java`
- **优先级**：P1 ｜ **状态**：待评估

### G4 · 无密码策略

- **现象**：`create/resetPassword` 直接 `encode(raw)`，无复杂度/长度/历史/强制改密。
- **已落地（初步强度校验）**：新增自定义校验注解 `@StrongPassword`（`common/validation/`），规则：长度 8-64、至少同时含字母与数字、不含空白；空值放行交由 `@NotBlank`。挂到所有「设置密码」入口——`UserCreateDTO.password`（建用户）、`ResetPasswordDTO.password`（管理员重置）、`ChangePasswordDTO.newPassword`（自助改密）。校验失败由 `GlobalExceptionHandler` 统一转 400 中文消息。规则常量集中于 Validator，留好后续配置化扩展位。`PlatformSeeder` 直接 `encode` 落库不走 DTO 校验，默认超管密码不受影响。前端表单同步加同规则校验。
- **剩余（仍 P1）**：密码历史、有效期、首次/重置后强制改密、强度规则配置化。
- **涉及**：`common/validation/StrongPassword.java`+`StrongPasswordValidator.java`、`UserCreateDTO`/`ResetPasswordDTO`/`ChangePasswordDTO`、前端 `UserList.vue`/`Profile.vue`
- **优先级**：P1 ｜ **状态**：已完成（强度校验；历史/有效期/强制改密按需另排）

### G5 · 会话策略偏宽 + 状态变更不失效会话 ✅

- **现象**：`timeout=30天`、`active-timeout=-1`（永不闲置失效）；租户停用、套餐变更后既有在线会话不强制失效（权限实时跟随，但登录态仍在）。
- **已落地**：
  - **收紧会话策略**：`application.yml` 将 `active-timeout` 由 `-1` 改为 `7200`（闲置 2 小时无操作自动失效，Sa-token 每次请求自动续期活跃时间）；`timeout` 维持 30 天（按产品定位，可后续在 profile 覆盖）。
  - **用户级会话主动失效**：`UserService` 在三处状态变更后调 `StpUtil.kickout(userId)` 踢出该用户全部在线会话——停用用户（`update` 中 `status==0`）、删除用户（`delete`）、管理员重置密码（`resetPassword`，强制重新登录）。`userId` 为全局唯一雪花，按 loginId 踢出无需租户上下文；`is-concurrent=true` 已满足。
  - **租户停用 / 过期**：由 G11 的 `TenantStatusInterceptor` 每次请求现查租户状态，`status==0` 或过期即 403 拦截（惰性、只读不写），既有会话登录态虽在但访问被挡，续费/恢复后下次请求自动放行。
  - **套餐变更**：`StpInterfaceImpl` 每次鉴权实时读 `package_id` + `renew()` 调 `evictTenant` 清缓存，边界实时生效，无需踢会话。
- **涉及**：`application.yml`、`system/service/UserService.java`、（已有）`TenantStatusInterceptor`/`StpInterfaceImpl`/`TenantService.renew`
- **优先级**：P1 ｜ **状态**：已完成

## 四、SaaS 核心能力

### G8 · 套餐只圈菜单边界，无配额限额 ✅

- **现象**：`sys_package` 仅关联菜单，无用户数/存储/调用量等配额，租户可无限建用户。
- **方案（已落地）**：配额采用「字典 + 引用」两表模型——`sys_quota_def(quota_key 唯一, name, remark, sort)` 定义配额项（含前端展示名称），`sys_package_quota(package_id, quota_id, quota_value)` 引用字典记录各套餐上限；`-1`/无记录=不限（缺省放行）。基座内置定义 `max_users → 最大用户数`（`QuotaKeys.MAX_USERS`），种子：基础版 10、全功能不限。通用 `QuotaService.checkAndAssert(key, intendedTotal)`（按 key 查 def → 查套餐值）在 `UserService.create` 拦截。**新增配额维度（含 biz 模块）只需向 `sys_quota_def` 注册定义 + 在写操作处调 `QuotaService`，无需改基座代码**。前端套餐页配额从表格列移除，改为操作列「配置配额」按钮 + 独立对话框（按名称展示、空=不限），权限码 `system:package:quota`。
- **演进**：初版为键值表 `sys_package_quota(quota_key, quota_value)` + 套餐 DTO 内联 `Map<String,Long> quotas`（`V4`）。因配额维度可能很多、且需中文名称展示，`V5__quota_def.sql` 重构为字典引用模型并将配额配置从套餐增改表单中拆出为独立接口（`GET/PUT /system/packages/{id}/quotas`）。
- **涉及**：Flyway `V4__package_quota.sql`/`V5__quota_def.sql`、`TenantLineHandlerImpl`(IGNORE_TABLES)、`SysQuotaDef`+Mapper、`SysPackageQuota`+Mapper、`QuotaKeys`、`QuotaService`、`UserService`、`PackageController/Service`、`AssignQuotasDTO`/`PackageQuotaVO`、前端 `package.ts`/`PackageList.vue`
- **优先级**：P1 ｜ **状态**：已完成

### G11 · 无订阅生命周期

- **现象**：`tenant.package_id` 永久绑定，无有效期/到期/计费/续费。
- **已落地**：
  - 接通悬空的 `tenant.expire_at`——`TenantCreateDTO/Service/VO` 支持设置/展示到期时间。
  - **续费 + 套餐变更**：`PUT /system/tenants/{id}/renew`（超管）覆盖到期时间（留空=永久，用 update wrapper 显式 set 以落库 null），可选 `changePackageId` 同时升/降套餐（套餐边界由 `StpInterfaceImpl` 每次鉴权实时读 `package_id`，变更即生效，无需踢会话）。前端「续费」对话框含套餐下拉。
  - **惰性过期判定（不落库、无定时任务）**：`TenantService.isExpired(tenant)` 共享判定逻辑（平台租户 + 永久豁免）。请求级 `TenantStatusInterceptor`（注册于登录拦截器之后）对已登录会话校验「租户存在 / 停用 / 过期」，过期或停用即拦（403），续费后下次请求自动恢复。`AuthService.login` 复用同一判定。**只读不写**——不修改 `tenant.status`，避免审计污染与状态漂移。
  - **过期展示**：`TenantVO.expired` 派生字段（现算），前端列表与续费框「已过期」红标。
- **剩余（仍 P2）**：独立订阅记录（起止/状态/历史）、临期预警、计费对接（计费建议移交上层 `biz/billing`，基座不实现）。
- **涉及**：`Tenant`/`TenantCreateDTO`/`TenantRenewDTO`/`TenantController`/`TenantService`/`TenantVO`/`AuthService`/`TenantStatusInterceptor`/`SaTokenConfig`、前端 `tenant.ts`/`TenantList.vue`
- **优先级**：P2 ｜ **状态**：部分完成

### G10 · 鉴权多次查库且无缓存 ✅

- **现象**：`StpInterfaceImpl` 每次鉴权发 5~6 条查询，无缓存。
- **方案（已落地）**：
  - **通用缓存能力**：引入 `spring-boot-starter-cache`，`config/CacheConfig` 启用 `@EnableCaching` 并配置 Redis 后端 `RedisCacheManager`（Jackson 3 `GenericJacksonJsonRedisSerializer` 序列化值，key 前缀 `bedrock:cache:` 与 Sa-token 会话隔离，默认 TTL 30 分钟）。基座/业务模块此后可直接用 `@Cacheable`/`@CacheEvict`。
  - **鉴权缓存**：把原 `StpInterfaceImpl` 的查库与裁剪逻辑下沉到 `system/service/PermissionCacheService`，`permissions/roles` 两方法加 `@Cacheable`，缓存 key = `tenantId:userId`，TTL 10 分钟。`StpInterfaceImpl` 退化为「解析会话租户/用户 → 委托缓存服务」的薄壳。命中缓存即省去 5~6 条查询。
  - **精确失效（按粒度，不一刀切）**：缓存 key 含租户前缀，故可按粒度清除——用户角色分配/删除→只清该用户（`evictUser`）；角色菜单分配/删除→只清当前租户（`evictTenant`，按 `tenantId:*` 前缀 SCAN 删除）；套餐菜单变更/删除→只清订阅该套餐的那些租户（`evictByPackage`）；租户换套餐→只清该租户；仅「全局菜单 perm 变更」才整体清空（`evictAll`，本就影响所有租户）。规避了「改一个租户套餐却清空全部租户缓存」的问题。
- **涉及**：`pom.xml`、`config/CacheConfig`、`system/StpInterfaceImpl`、`system/service/PermissionCacheService`、失效接入 `UserService`/`RoleService`/`PackageService`/`TenantService`/`MenuService`
- **优先级**：P2 ｜ **状态**：已完成

### G12 · 缺自助能力

- **现象**：无注册/自助开通租户、无找回密码、无邮箱/手机验证、无"修改自身密码/个人中心"（`resetPassword` 仅管理员重置）。

  - `sys_user` 加 `nickname`/`phone` 两列（`V7__sys_user_profile.sql`，均可空、不强制唯一）。
  - 三个「当前登录用户」接口（`UserController`，登录即可、无额外权限码）：`GET /system/users/me/profile`（查资料）、`PUT /system/users/me/profile`（改昵称/手机）、`PUT /system/users/me/password`（改自身密码，先校验原密码、拒绝新旧相同）。`me` 为字面量子路径，与 `{id}` 不冲突。

- **明确不做**：自助注册、找回密码、邮箱/手机验证（无邮件/短信基础设施）；头像上传（无文件存储服务）。如面向公开 SaaS 需要，另行排期。
- **涉及**：`V7__sys_user_profile.sql`、`SysUser`、`UpdateProfileDTO`/`ChangePasswordDTO`、`ProfileVO`/`MeVO`/`UserVO`、`UserController`/`UserService`、`AuthService`、前端 `api/user.ts`/`api/auth.ts`/`Profile.vue`/`UserList.vue`
