# 项目协作平台 — 技术方案

> 版本：v0.4（待评审）  日期：2026-06-08
> 技术栈：Vue3 前端 + Spring Boot 后端 + PostgreSQL（MyBatis-Plus 多租户）+ Sa-token 鉴权

---

## 1. 项目定位与范围

系统定位为 **多租户项目协作平台**，以项目协同为核心。共 3 大业务域：

| # | 业务域 | 核心能力 |
|---|--------|----------|
| 1 | 用户与权限 | 注册/登录、Sa-token 鉴权、RBAC 角色权限、多租户隔离 |
| 2 | 主数据管理 | 企业信息、人员（联系人）、相关方（客户/供应商/合作伙伴） |
| 3 | 项目进度管理 | 项目立项、里程碑、进度跟踪、甘特/看板视图 |

> **概念区分**：**人员（`person`）** 是协作过程中涉及的人（如联系人、项目成员），不一定有登录账号；**系统用户（`sys_user`）** 是能登录系统的账号，可绑定到某个人员。二者一对一可选关联。

> ⚠️ **范围提醒**：建议**分期交付**（见第 9 节），先夯实用户权限 + 主数据，再叠加项目进度。

## 2. 总体架构

```
┌─────────────────────────────────────────────────────┐
│                     浏览器 (SPA)                       │
│   Vue3 + Vite + TypeScript + Pinia + Vue Router       │
│   Element Plus (UI) + Axios + ECharts(图表/甘特)       │
└───────────────────────────┬─────────────────────────┘
                            │ HTTPS / REST（携带 Sa-token 令牌）
┌───────────────────────────┴─────────────────────────┐
│                  Spring Boot 后端 (单体分层)           │
│  Controller → Service → Mapper(MyBatis-Plus)          │
│  Sa-token（登录态/注解鉴权） | 全局异常 | 参数校验      │
│  多租户插件 TenantLineInnerInterceptor 自动隔离        │
│  模块化包结构：auth / org / project                   │
└───────────────────────────┬─────────────────────────┘
                            │ JDBC
┌───────────────────────────┴─────────────────────────┐
│                      PostgreSQL                       │
│   Flyway 管理 schema 版本  |  逻辑外键 + 索引          │
└───────────────────────────┬─────────────────────────┘
                            │
                       Redis（Sa-token 会话 / 缓存）
```

**架构选型说明**：先采用**模块化单体（Modular Monolith）**，按业务域分包，边界清晰但部署简单。规模增长后可平滑拆分为微服务。不一上来就上微服务，避免过度设计。

## 3. 技术栈明细

### 3.1 前端
| 类别 | 选型 | 说明 |
|------|------|------|
| 框架 | Vue 3 (Composition API) | `<script setup>` 语法 |
| 构建 | Vite 5 | 快速冷启动与 HMR |
| 语言 | TypeScript | 类型安全 |
| 状态 | Pinia | 官方推荐，替代 Vuex |
| 路由 | Vue Router 4 | 路由级权限守卫 |
| UI 库 | Element Plus | 中后台组件齐全 |
| HTTP | Axios | 统一拦截器处理 token/错误 |
| 图表 | ECharts | 甘特图、项目进度图 |
| 工具 | dayjs / lodash-es | 日期与工具函数 |
| 代码规范 | ESLint + Prettier | 统一风格 |

### 3.2 后端
| 类别 | 选型 | 说明 |
|------|------|------|
| 框架 | Spring Boot 3.x | 需 JDK 17+ |
| Web | Spring Web (REST) | |
| 安全 | **Sa-token** | 登录认证 + 注解式角色/权限校验；会话存 Redis（`sa-token-redis-jackson`）分布式共享；不依赖 Spring Security / JWT |
| 持久层 | **MyBatis-Plus** | SQL 可控、代码生成器、内置分页与多租户插件 |
| 多租户 | **MyBatis-Plus `TenantLineInnerInterceptor`** | 自动按 `tenant_id` 改写 SQL，业务代码无感 |
| 数据库迁移 | Flyway | schema 版本化，启动时自动按版本顺序执行 SQL，不重复 |
| 校验 | Jakarta Validation | `@Valid` 参数校验 |
| 文档 | SpringDoc OpenAPI (Swagger UI) | 自动生成 API 文档 |
| 工具 | MapStruct / Lombok | Entity/DTO/VO 映射、样板代码 |
| 构建 | Maven | 默认构建工具 |

### 3.3 基础设施
- **数据库**：PostgreSQL 15+
- **缓存/会话**：Redis 7+ — 存储 Sa-token 登录会话（分布式）、热点缓存、临期提醒辅助
- **容器化**：Docker + docker-compose 一键拉起 PostgreSQL + Redis + 后端 + 前端，区分开发与生产环境
- **多环境**：开发(dev) 与 生产(prod) 两套 — 后端 Spring Profiles（dev/prod）+ 前端 Vite 模式（.env.development / .env.production）。开发测试统一用 dev（跑在测试机），prod 部署到云服务器
- **定时任务**：Spring `@Scheduled` 或 Quartz（里程碑临期提醒扫描，可选）

## 4. 数据模型（核心实体）

> 采用 PostgreSQL，统一字段：`id (bigserial)`、`tenant_id (租户隔离)`、`created_by`、`created_at`、`updated_by`、`updated_at`、`deleted (软删除)`。
>
> **多租户策略**：`tenant` 是顶层租户（系统订阅方），与业务实体 `enterprise`（租户内管理的企业数据）区分开。除 `tenant` 表本身外，所有业务表均带 `tenant_id`，引用 `tenant.id`。MyBatis-Plus 的 `TenantLineInnerInterceptor` 会在 SQL 执行时自动追加 `WHERE tenant_id = ?`，从 Sa-token 当前登录会话（登录时写入的 `tenant_id`）取值，业务代码无需手动过滤。

### 4.1 租户与权限
- `tenant`：租户（系统订阅方）（name, code, status, expire_at, contact） — **不带 `tenant_id`，是隔离的根**
- `sys_user`：系统用户（username, password_hash, status, **person_id 可选绑定人员**）
- `sys_role`：角色（code, name）
- `sys_permission`：权限（code, name, type）
- `sys_user_role` / `sys_role_permission`：多对多关联

### 4.2 主数据（组织域 org）
- `enterprise`：企业（租户内管理的企业业务数据）（name, unified_code 统一社会信用代码, address, industry, contact）
- `person`：人员（协作涉及的人/联系人，**不要求有登录账号**）（name, phone, email, enterprise_id, status）
- `stakeholder`：相关方（项目签约/往来方，**可能是企业也可能是个人**）（relation_type[客户/供应商/合作伙伴], party_type[企业/个人], party_id → 多态指向 `enterprise.id` 或 `person.id`, contact_info）

### 4.3 项目域 project
- `project`：项目（name, code, status, owner_id, start_date, end_date, progress, enterprise_id）
- `milestone`：里程碑（project_id, name, due_date, status）
- `project_member`：项目成员（project_id, person_id, role）

**ER 关系要点**：`tenant` 是隔离根，所有业务数据按 `tenant_id` 归属；`sys_user` 可选绑定 `person`；`person` 关联项目成员；`project` 关联里程碑与成员；`stakeholder` 多态指向 `enterprise` 或 `person`；`enterprise` 是租户内的企业业务实体。

## 5. API 设计（REST 约定）

- 统一前缀：`/api/v1`
- 统一响应体：`{ code, message, data }`
- 鉴权：除登录/注册外，请求头携带 Sa-token 令牌（默认头名 `satoken`，可配置为 `Authorization`）
- 分页：`?page=1&size=20&sort=createdAt,desc`

示例接口（节选）：
```
POST   /api/v1/auth/login           登录，返回 token
POST   /api/v1/auth/register        注册
GET    /api/v1/persons              人员列表
GET    /api/v1/enterprises          企业列表
```
完整接口由 Swagger UI 自动维护。

## 6. 权限模型（RBAC）

预设角色（均限定在所属 `tenant` 内）：
| 角色 | 权限范围 |
|------|----------|
| 超级管理员 | 跨租户全部（平台运维） |
| 租户管理员 | 本租户所有数据（用户、主数据、项目等） |
| 普通用户 | 本人参与的项目等相关数据 |

- 后端：Sa-token 注解 `@SaCheckRole` / `@SaCheckPermission` 做方法级校验；租户级隔离由 MyBatis-Plus 多租户插件自动完成，业务级再按 `owner_id` 等过滤。
- 前端：路由守卫 + 按钮级权限指令 `v-permission`。

## 8. 非功能性需求

- **安全**：密码 BCrypt 加盐；接口防越权；多租户严格隔离。
- **性能**：列表接口分页 + 索引；高频读可加 Redis 缓存。
- **可观测**：统一日志（logback）、全局异常处理、操作审计表。
- **测试**：后端 JUnit + MockMvc；前端 Vitest（关键逻辑）。

## 10. 部署与环境

### 10.1 环境划分（两套）
| 环境 | 用途 | 载体 | Spring Profile | 前端模式 |
|------|------|------|----------------|----------|
| 开发 dev | 开发 + 测试 | 测试机 | dev | development |
| 生产 prod | 线上 | 云服务器 | prod | production |

### 10.2 后端多环境
- `application.yml` 放公共配置；`application-dev.yml` / `application-prod.yml` 放各环境差异（DB、Redis 地址、日志级别、Sa-token 配置）。
- 通过环境变量 `SPRING_PROFILES_ACTIVE=dev|prod` 激活。
- 生产敏感配置（DB/Redis 密码等）通过环境变量注入，不写入仓库。

### 10.3 前端多环境
- Vite 按模式加载 `.env.development` / `.env.production`，区分 `VITE_API_BASE_URL` 等。
- 生产构建：`vite build --mode production`，产物由 Nginx 容器托管。

### 10.4 一键部署
- 基础编排 `docker-compose.yml` 定义 postgres + redis + backend + frontend(nginx) 四个服务。
- 环境覆盖：`docker-compose.dev.yml`（暴露端口、挂本地卷、热重载）与 `docker-compose.prod.yml`（资源限制、健康检查、不对外暴露 DB/Redis）。
- 启动命令：
  - 测试机（开发/测试）：`docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d`
  - 云服务器（生产）：`docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d`
- 建议用 Makefile 封装为 `make up-dev` / `make up-prod`，做到真正"一键"。
