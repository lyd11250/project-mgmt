# 业务模块接入规范

> 本文件是**上层业务模块（`biz/*`）接入基座（Bedrock）的标准范式**。新增任何业务模块前先通读本文件，并照 `biz/party` 样板落地。
> 基座规范见 [CLAUDE.md](CLAUDE.md)，系统能力设计见 [功能设计.md](功能设计.md)（第 3 节为接入约定总览）。
> 活样板：`party`（相关方主数据）——首个业务模块，本文件以它为例贯穿说明。

---

## 0. 基本原则

- **基座不内置业务功能**。业务模块以「配置/套餐驱动」方式接入：单一代码库统一开发，某租户「能用哪些功能」由**套餐裁剪**决定，不为单租户写专属代码。
- **完全复用基座机制**：统一响应（`Result`）、全局异常（`BusinessException` + `GlobalExceptionHandler`）、多租户隔离、审计自动填充、RBAC 鉴权、雪花主键、软删除——业务模块**不重复造**，直接沿用。
- **边界纪律**：业务模块经基座 service 接口交互，**不跨模块直连他人 mapper**，以保留将来按模块切微服务的可能。

---

## 1. 代码位置与包结构

基础包 `com.github.lyd11250.bedrock`，业务模块统一在 `biz.<module>` 下，分层与命名仿基座 `system` 域：

```
biz/<module>/
  controller/   接口层，返回 Result<T>，@SaCheckPermission 校验
  service/      业务逻辑，贫血模型，DTO/VO/Entity 手动互转
  mapper/       MyBatis-Plus BaseMapper
  entity/       持久层对象，@TableName 指定表名，继承 BaseEntity
  dto/          接口入参（xxxDTO）
  vo/           接口出参（xxxVO）
  package-info.java  模块简介
```

> 样板：`biz/party`（`Party`/`PartyPerson`/`PartyOrganization` 三实体 + Person/Organization 两套 controller/service）。
> Mapper 由 `BedrockApplication` 的 `@MapperScan("com.github.lyd11250.bedrock.**.mapper")` 自动扫描，无需额外配置。

## 2. 数据库与多租户

- **迁移脚本同序列递增**：业务表 DDL 写在 `backend/src/main/resources/db/migration/`，文件名 `V{n}__{描述}.sql`，版本号在基座现有序列上继续递增（与 `system` 共用一套 `flyway_schema_history`），**不修改已合并的历史脚本**。
- **业务表必须带 `tenant_id`**（除非确实全局共享），由多租户插件自动追加 `WHERE tenant_id = ?` 并在 insert 时自动写入——**业务代码无需手动过滤/赋值租户**。不参与隔离的全局表才加入 `TenantLineHandlerImpl.IGNORE_TABLES`。
- **建表约定**：主键 `BIGINT`（应用层雪花算法）；统一审计列 `created_by/at、updated_by/at` + 软删除列 `deleted SMALLINT 0/1`；唯一索引带 `WHERE deleted = 0` 偏过滤（可空字段再加 `AND col IS NOT NULL`）；表/列加 `COMMENT`。
- **实体**继承 `common.BaseEntity`（含 `id/tenantId/审计/deleted`），子类加 `@EqualsAndHashCode(callSuper = true)`，**不重复声明公共字段**。全局表（无 `tenant_id`）继承 `common.GlobalBaseEntity`。

> 样板 `V9__party_module.sql`：`party`（主表）+ `party_person`/`party_organization`（子表），三表均带 `tenant_id`；`id_card`/`tax_no` 用 `WHERE deleted=0 AND col IS NOT NULL` 偏过滤唯一索引。

### 关于「共享主键」（子类型 1:1 建模）
当一个实体需要按子类型拆表（如相关方 = 人员 | 组织），采用 **Fowler Party 模式**：主表存公共身份，子表存类型专属字段，**子表 id = 主表 id 共享主键**。实现要点：子表实体不重声明 `id`（沿用 `BaseEntity` 的 `ASSIGN_ID`），创建时先插主表拿到雪花 id，再 `setId(主表id)` 插子表——`ASSIGN_ID` 在 id 已赋值时沿用、不再生成。其他模块统一以主表 id 引用。

## 3. 鉴权与菜单注册（DB 驱动，三步接入）

权限码即全局菜单 `sys_menu.perm`，格式 `模块:资源:动作`（如 `party:person:create`）。新增功能按三步接入：

1. **注册菜单 perm**：在业务模块的 Flyway 脚本中 `INSERT INTO sys_menu`，登记目录（M）/页面（C，带 `path`+`component`）/按钮（F，带 `perm`）。`component` 指向前端组件标识（如 `party/PersonList`，对应 `views/party/PersonList.vue`），由前端动态路由按名加载。菜单 id 用固定小 `BIGINT`（避开系统菜单 1xx 与雪花大 id）。
2. **纳入套餐**：`INSERT INTO sys_package_menu` 把菜单纳入相应套餐（哪些租户可用由套餐决定）。样板把 party 菜单纳入「全功能（FULL，id=2）」套餐；「基础版（BASIC）」不含，以体现套餐裁剪。`SeedService` 按 `sys_package_menu` 动态播种角色菜单，**无需改动**。
3. **接口挂权限码**：Controller 方法加 `@SaCheckPermission("模块:资源:动作")`；开放接口（极少）需在 `SaTokenConfig.EXCLUDE_PATHS` 显式放行。

> 鉴权判定（`StpInterfaceImpl`）：用户权限 = 角色分配的菜单 perm ∩ 租户套餐边界；超管（`SUPER_ADMIN`）通配 `*` 绕过边界。业务模块无需关心，只管挂 perm。

## 4. API 与接口规范

- **URL `/api/v1/{module}/{resource}`**：业务模块用各自模块名前缀（如 `/api/v1/party/persons`、`/api/v1/party/organizations`），与 `biz/*` 包边界对齐。资源用复数名词，动作语义靠 HTTP 方法，不在路径塞动词；子资源走层级（`/{id}/xxx`）。
- **统一响应**：Controller 一律返回 `Result<T>`（`Result.ok(data)` / `Result.ok()`）；业务错误 `throw new BusinessException("…")`，由全局处理器转 `{code,message,data}`，**不手工 new 错误 Result、不在 Controller 内 try-catch**。
- **入参校验**：DTO 用 Jakarta Validation（`@NotBlank` 等，中文消息）+ Controller `@Valid`。
- **分页**：`Page.of(current,size)` + `Wrappers.lambdaQuery()`，返回 `IPage<VO>`（`page.convert(this::toVO)`）。
- **DTO/VO/Entity 三分**，Service 内手动互转（参 `PersonService.toVO`），不在 Entity 上直接出入参。
- **API 文档**：Controller 加 `@Tag(name="…")` 归类 Swagger；dev 开放 `/swagger-ui.html`，prod 关闭。

## 5. 前端接入（三件套，零 router 改动）

前端页面由后端菜单树**动态挂载**（`sys_menu.component` 字符串 → `views/<component>.vue`），新增页面无需手改 `router/index.ts`。三件套：

1. **api 层** `src/api/<module>.ts`：每域一文件，函数式封装 `request.get/post/...`；响应已被拦截器解包，调用方直接拿 `data`；分页返回 `PageResult<T>`。
2. **视图** `src/views/<module>/*.vue`：按域分目录，组件路径与菜单 `component` 对齐；用 Element Plus；按钮级权限 `v-permission="'模块:资源:动作'"`；时间显示用 `formatDateTime()`，日期选择器配 `format`/`value-format`。
3. **菜单**：由第 3 节的 `sys_menu` 种子下发，自动出现在侧边导航与动态路由——前端无需登记。

> 样板：`api/party.ts` + `views/party/{PersonList,OrganizationList}.vue`。组织类型用 `el-autocomplete` 拉 `/party/organizations/types` 历史值补全；成立日期用 `el-date-picker`（`format/value-format=YYYY-MM-DD`）。

## 6. 自测与提交

- **后端**改动后至少本机跑通 `cd backend && ./mvnw -s settings.xml -DskipTests compile`。
- **前端**改动后跑通 `cd frontend && npm run build`。
- 本机无 Docker：Flyway 迁移与页面交互需在**测试机**实跑验证（三视角：超管/租户管理员/普通用户）。
- 提交遵循 Conventional Commits（中文标题）；业务模块的设计文档与进度**就近维护**（不并入基座 `开发进度.md`）。
