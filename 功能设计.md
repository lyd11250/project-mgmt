# Bedrock 基座功能设计

> 本文件记录 **SaaS 平台基座（Bedrock）的系统功能设计**（做什么、业务规则如何）。
> 工程层面（技术选型/架构/部署）见 [技术方案.md](技术方案.md)。
> **范围**：仅系统能力（租户/套餐/RBAC/菜单/鉴权）。上层租户业务功能不在此文件，其设计文档随业务模块就近存放（见第 3 节）。
> 最近更新：2026-06-09

---

## 1. 业务域总览

系统按**两类业务**组织，二者正交：

| 类别 | 含义 | 归属 |
|------|------|------|
| **系统能力（基座）** | SaaS 必备：租户、套餐、RBAC（用户/角色/菜单）、鉴权、多租户隔离 | `system` |
| **业务定制功能** | 面向租户的实际业务功能 | `biz/*`（上层业务模块，基座不内置） |

**本文件只覆盖系统能力**；业务定制功能各自随模块就近设计（见第 3 节）。

**关键原则（配置/套餐驱动 SaaS）**：单一代码库，功能统一开发；某租户"能用哪些功能"由**套餐裁剪**决定，**不为单个租户写专属代码**。所谓"定制"是全量功能的**子集开关**，而非分叉实现。

用两个正交的轴看更清楚——"系统能力 vs 业务功能" × "平台维护 vs 租户维护"：

| | **平台维护**（超管） | **租户维护**（租户管理员） |
|---|---|---|
| **系统能力** | 全局菜单目录定义、套餐定义、租户管理 | 本租户用户、角色、角色↔菜单分配 |
| **业务功能** | 功能模块的统一开发 | 在套餐范围内使用功能、产生业务数据 |

> 易错点：菜单的**「定义」是平台的事**（全局一份），菜单的**「分配」是租户的事**（租户内 RBAC）。两者分属不同层。

---

## 2. 系统业务域：RBAC 套餐模型

采用主流 SaaS 的**「全局菜单目录 + 套餐订阅 + 租户内 RBAC」三层模型**。

### 2.1 三层结构

```
① 全局菜单目录   平台定义全量菜单树（含按钮级权限码）          —— 全局唯一
② 套餐 package   套餐 = 菜单子集（基础版/专业版…）             —— 平台定义
③ 租户订阅       tenant.package_id 绑定套餐 → 圈定可用功能边界  —— 建租户时选
④ 租户内 RBAC    角色→分配菜单→给用户（仅限套餐范围内）         —— 租户管理员维护
```

### 2.2 数据模型

**全局层**（平台维护，**不带 `tenant_id`**，纳入多租户插件 `IGNORE_TABLES`）

| 表 | 关键字段 | 说明 |
|----|----------|------|
| `sys_menu` | id, parent_id, type, name, path, component, icon, perm, sort, visible, status | 菜单树。菜单与权限**合并为一棵树** |
| `sys_package` | id, name, code, status, remark | 套餐 |
| `sys_package_menu` | id, package_id, menu_id | 套餐 ↔ 菜单（多对多） |

`sys_menu.type`：`M` 目录（仅导航分组）/ `C` 菜单（对应页面路由）/ `F` 按钮（挂 `perm` 权限码，无路由）。
`perm` 命名约定：`模块:资源:动作`，如 `system:user:create`、`org:enterprise:list`。

**租户绑定**：`tenant` 增列 `package_id`（订阅的套餐；平台租户可订阅"全功能套餐"或由超管通配绕过）。

**租户层**（带 `tenant_id`，租户管理员维护）

| 表 | 关键字段 | 说明 |
|----|----------|------|
| `sys_user` | username, password_hash, status, person_id | 沿用 |
| `sys_role` | code, name | 沿用 |
| `sys_user_role` | user_id, role_id | 沿用 |
| `sys_role_menu` | role_id, menu_id | **取代** `sys_role_permission`：角色↔菜单分配 |

> 废弃 `sys_permission` / `sys_role_permission`：权限不再每租户复制一份，而是引用全局 `sys_menu`。

### 2.3 菜单树示例

```
系统管理 (M /system)
├─ 用户管理 (C /system/users)        perm: system:user:list
│  ├─ 新增 (F)  system:user:create
│  ├─ 编辑 (F)  system:user:update
│  └─ 删除 (F)  system:user:delete
├─ 角色管理 (C /system/roles)        system:role:list
├─ 菜单管理 (C /system/menus)        system:menu:list      ← 仅平台超管
├─ 套餐管理 (C /system/packages)     system:package:list   ← 仅平台超管
└─ 租户管理 (C /system/tenants)      system:tenant:list    ← 仅平台超管
（以上为基座系统菜单示例；业务模块菜单由各 biz 模块在 sys_menu 种子中自行注册，命名 `<module>:<resource>:<action>`）
```

平台专属菜单（菜单/套餐/租户管理）只对超管可见——通过超管通配 `*` 实现，不放入普通套餐。

### 2.4 鉴权判定

```
角色码 = 用户 → sys_user_role → sys_role.code
权限码 = 用户 → 角色 → sys_role_menu → sys_menu.perm
         ∩  租户套餐边界（tenant.package_id → sys_package_menu → sys_menu.perm）
超管（平台租户）= 通配 *，绕过套餐边界与菜单分配
```

由 `StpInterfaceImpl` 提供给 Sa-token；方法级仍用 `@SaCheckPermission("...")` / `@SaCheckRole("...")`。

### 2.5 平台 vs 租户职责

| 动作 | 平台超管 | 租户管理员 | 普通用户 |
|------|:---:|:---:|:---:|
| 定义全局菜单 | ✅ | — | — |
| 定义套餐 | ✅ | — | — |
| 建租户并选套餐 | ✅ | — | — |
| 建角色 / 分配菜单 | — | ✅（套餐内） | — |
| 建用户 / 授角色 | — | ✅ | — |
| 使用功能、产生数据 | ✅ | ✅ | ✅（授权内） |

### 2.6 前端连带

菜单由后端动态下发（`/auth/me` 或 `/system/menus/nav`）→ **动态导航 + 动态路由**，取代当前 `router/index.ts` 写死的菜单与 `RbacConstants` 硬编码权限。按钮级仍用 `v-permission`。

## 3. 业务定制模块接入约定

基座**不内置**任何具体业务功能。上层业务模块以"配置/套餐驱动"方式接入：

- **代码位置**：`biz/<module>/`，分层与命名仿基座 `system` 域（controller/service/mapper/entity/dto/vo）。
- **设计文档就近**：每个业务模块在自身目录下维护设计文档（如 `biz/<module>/README.md` 或 `设计.md`），**不写入基座主文档**。
- **接入基座的步骤**：
  1. 在 `sys_menu` 种子中注册本模块的菜单与按钮权限码（命名 `<module>:<resource>:<action>`）；
  2. 把菜单纳入相应套餐（`sys_package_menu`），由套餐决定哪些租户可用；
  3. 业务表带 `tenant_id`，复用基座的多租户隔离、审计、统一响应、鉴权机制。
- **边界纪律**：业务模块经基座 service 接口交互，**不跨模块直连他人 mapper**，以保留将来按模块切出微服务的可能（见 [技术方案.md](技术方案.md) 架构选型）。

## 4. 与现状代码的差异（重构点）

当前第 1 期-A 是「角色→权限码、每租户复制权限目录」的简化 RBAC，落地本模型需重构：

| 现状 | 目标 |
|------|------|
| `sys_permission` 带 `tenant_id`，每租户复制 | `sys_menu` 全局唯一，套餐订阅 |
| `RbacConstants` 硬编码权限码 | Flyway 初始化菜单数据，DB 驱动 |
| `sys_role_permission` | `sys_role_menu` |
| `SeedService.seedTenant` 复制权限目录 | 建租户绑套餐，只播角色+管理员 |
| `StpInterfaceImpl` 角色→权限 | 角色→菜单 perm ∩ 套餐边界 |
| `auth` 包 | 重组为 `system` 包 |
| 前端写死菜单/路由 | 后端下发，动态菜单/路由 |

> 该重构的任务拆解与排期见 [开发进度.md](开发进度.md)。

## 5. 待定 / 开放问题

- 套餐与功能的计费关系（是否含到期、用量限制）——暂不实现，预留 `tenant.expire_at`。
- 数据权限范围（行级，如"只看本人/本部门数据"）——当前仅到功能级，后续按需引入。
- 是否需要多组织/部门层级（租户内再分部门）——暂不纳入，留待业务确认。
